# 🤍 CONTRIBUTING

# 開発ガイドライン

このドキュメントは、推しメモを継続して開発するためのルールをまとめたものです。

推しメモでは、新しい機能を追加することよりも、
「推し活をもっと快適にすること」を最優先に開発を進めます。

---

# 基本方針

開発で迷った場合は、次の優先順位で判断します。

1. ユーザーが迷わないこと
2. 忘れないこと
3. 操作しやすいこと
4. 保守しやすいこと
5. 新しい機能

---

# 設計フロー

実装前に設計を行います。

基本的な流れ

1. PRODUCT
2. DESIGN
3. DATA
4. SPEC
5. ARCHITECTURE
6. COMPONENTS
7. 実装

設計書と実装は常に一致させます。

---

# 実装ルール

- コンポーネントは単一責任を意識する
- 共通化できるものは再利用する
- 同じデータを複数管理しない
- 状態管理はできるだけシンプルにする
- 命名は分かりやすく統一する

---

# UIルール

UIは「かわいい」よりも「使いやすい」を優先します。

判断に迷った場合は、

- 情報量
- 見やすさ
- 操作回数

が少なくなる設計を採用します。

---

# 機能追加の考え方

新機能を追加する前に、次の3つを確認します。

- 本当に必要な機能か
- 推し活で実際に役立つか
- 既存機能で代替できないか

必要以上に機能を増やさず、
シンプルなアプリを目指します。

---

# ドキュメント

仕様変更を行った場合は、
必要に応じてドキュメントも更新します。

- PRODUCT
- DESIGN
- DATA
- SPEC
- ARCHITECTURE
- COMPONENTS
- ROADMAP
- CHANGELOG

---

# 開発ポリシー

推しメモは、

**「推し活で『あ、忘れてた…』をなくすアプリ。」**

というコンセプトを軸に開発を続けます。

すべての機能は、このコンセプトに沿って判断・設計・実装します。
